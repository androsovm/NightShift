# NightShift — Идеи по улучшению

Анализ проекта на 2026-03-22. Группировка по категориям, размер оценён как S (< 1 дня), M (1–3 дня), L (3–7 дней), XL (> 1 недели).

---

## 1. Надёжность (Reliability)

### 1.1 File locking для tasks.yaml — S
**Проблема:** `task_queue.py` делает атомарный `os.replace`, но не использует file lock. Если два процесса (например, `nightshift run` и `nightshift sync`) работают одновременно, один перезапишет изменения другого — classic lost update.
**Решение:** Использовать `fcntl.flock` (Unix) или `portalocker` для advisory lock при чтении/записи `tasks.yaml`.

### 1.2 Graceful shutdown и SIGTERM handling — M
**Проблема:** Если `nightshift run` убивается (OOM, kill, перезагрузка), задача остаётся в статусе `pending`, а в репо может быть грязное состояние (незакоммиченная ветка, uncommitted changes).
**Решение:** Signal handler для SIGTERM/SIGINT: пометить текущую задачу как `failed`, вернуть репо на main, сохранить partial result.

### 1.3 Retry с exponential backoff — S
**Проблема:** Retry delay фиксированный — 30 секунд (`_RETRY_DELAY_SECONDS = 30`). При длительных outage'ах 3 попытки по 30 секунд недостаточно.
**Решение:** Экспоненциальный backoff: 30s → 60s → 120s. Можно также увеличить `_MAX_RETRIES` до 5 для сетевых ошибок.

### 1.4 Проверка чистоты рабочего дерева перед стартом — S
**Проблема:** `prepare_repo` делает `fetch + checkout main + pull`, но не проверяет, нет ли в репо uncommitted changes или stashed changes от предыдущего прерванного запуска.
**Решение:** Добавить `git status --porcelain` + `git stash` или `git reset --hard` с логированием.

### 1.5 Heartbeat / watchdog — M
**Проблема:** Если Claude зависает или subprocess.run не возвращает управление (edge case), нет внешнего механизма для обнаружения зависших запусков.
**Решение:** Писать heartbeat-файл (`~/.nightshift/heartbeat`) каждые N минут. Отдельный `nightshift watchdog` или проверка в `doctor`.

### 1.6 Fallback основной ветки — S
**Проблема:** `prepare_repo` и `cleanup_branch` хардкодят `main`. Некоторые репо используют `master`, `develop` или другие ветки.
**Решение:** Автодетект через `git symbolic-ref refs/remotes/origin/HEAD` или конфигурация `default_branch` в `.nightshift.yaml`.

### 1.7 Повторные попытки для failed-задач — M
**Проблема:** Если задача завалилась, она переходит в `failed` и больше не подбирается автоматически. Нет механизма auto-retry на следующую ночь.
**Решение:** Конфигурируемый `max_attempts` (default: 1). Если attempts < max_attempts, задача остаётся `pending`. Также: cooldown между попытками (не ретраить в ту же ночь).

## 2. Безопасность (Security)

### 2.1 Валидация промпта от внешних источников — M
**Проблема:** Issue body из GitHub/YouTrack/Trello подставляется прямо в промпт для Claude (`task.intent = issue.get("body")`). Злоумышленник может создать issue с prompt injection, заставив Claude выполнить произвольные команды.
**Решение:** Sanitization: стриппить code blocks с shell/bash, ограничивать длину, добавить в system prompt чёткие запреты. Возможно — отдельный "review" проход, где Claude проверяет intent на безопасность перед выполнением.

### 2.2 Sandboxing выполнения Claude — L
**Проблема:** `--dangerously-skip-permissions` даёт Claude полный доступ к файловой системе. Если задача описана некорректно, Claude может удалить файлы, изменить конфиги вне проекта и т.п.
**Решение:** Исследовать запуск в docker-контейнере или с ограниченным PATH/chroot. Как минимум — проверять, что все изменённые файлы находятся внутри project_path.

### 2.3 Аудит лог действий — M
**Проблема:** Нет подробного лога того, какие файлы были изменены, какие команды выполнялись во время каждой задачи. Если что-то пошло не так — трудно расследовать.
**Решение:** Сохранять полный git diff для каждой задачи, список изменённых файлов, выход Claude. Структурированный отчёт в `~/.nightshift/audit/`.

### 2.4 Ротация и ограничение размера логов — S
**Проблема:** Логи в `~/.nightshift/logs/` и `~/.nightshift/runs/` растут бесконечно. На длительной дистанции — десятки гигабайт.
**Решение:** Конфигурируемый `log_retention_days` (default: 30). Команда `nightshift cleanup` или автоочистка при старте.

### 2.5 Проверка permissions на .env файл при каждом чтении — S
**Проблема:** `secrets.py` ставит chmod 600 при создании и записи, но не проверяет при чтении. Кто-то может случайно сделать файл world-readable.
**Решение:** При `load_secrets()` проверять `stat.S_IMODE` и предупреждать (или отказываться читать), если permissions слишком открытые.

## 3. Фичи (Features)

### 3.1 Нотификации о результатах — M
**Проблема:** После ночного запуска результат можно увидеть только через `nightshift status` или зайдя в GitHub. Нет push-уведомлений.
**Решение:** Плагинная система нотификаций: Telegram bot, Slack webhook, email, macOS notifications. Отправлять дайджест: N passed, M failed, ссылки на PR.

### 3.2 Веб-дашборд — XL
**Проблема:** Весь UI — CLI. Для быстрого обзора статуса проектов неудобно.
**Решение:** Лёгкий веб-сервер (FastAPI + htmx) с дашбордом: история запусков, графики, ссылки на PR. `nightshift dashboard` запускает на localhost.

### 3.3 Linear как источник задач — M
**Проблема:** Linear — популярный таск-трекер, но адаптера нет.
**Решение:** Реализовать `LinearSource` аналогично `GitHubSource`. Linear имеет удобный GraphQL API.

### 3.4 Jira как источник задач — M
**Проблема:** Jira — самый распространённый таск-трекер в enterprise.
**Решение:** Реализовать `JiraSource`. Jira REST API v3. Документировать как пример plugin-адаптера.

### 3.5 Параллельное выполнение задач для разных проектов — L
**Проблема:** Проекты обрабатываются последовательно. Если у пользователя 5 проектов по 3 задачи — всё выполняется линейно, что может занять всё ночное окно.
**Решение:** `asyncio.gather` или `ProcessPoolExecutor` для параллельной обработки разных проектов (не задач внутри одного проекта — они меняют один репо).

### 3.6 Поддержка monorepo — L
**Проблема:** Один проект = один git repo. В monorepo несколько "логических проектов" живут в одном репо, каждый со своим тестовым набором.
**Решение:** Опция `working_dir` в `.nightshift.yaml` для указания поддиректории. Quality gates запускаются внутри неё.

### 3.7 Custom quality gates — M
**Проблема:** Quality gates зашиты в код (blast radius, linter, tests). Нельзя добавить свою проверку (type checker, security scan, custom script).
**Решение:** Конфигурация `quality_gates` в `.nightshift.yaml` со списком shell-команд. Каждый gate: имя, команда, exit code 0 = pass.

### 3.8 Автоматическое закрытие issue после мержа PR — M
**Проблема:** По MVP человек закрывает задачу в источнике вручную. Это tedious.
**Решение:** GitHub webhook listener или периодическая проверка: если PR merged → вызвать `adapter.mark_done()`. Или использовать "Closes #N" в теле PR (для GitHub).

### 3.9 Поддержка нескольких LLM-провайдеров — L
**Проблема:** Жёсткая привязка к `claude` CLI. Нельзя использовать другие модели или API напрямую.
**Решение:** Абстракция `LLMExecutor` с реализациями: `ClaudeCLI`, `ClaudeAPI`, `OpenAI`, `Ollama`. Конфигурация в `.nightshift.yaml`.

### 3.10 Зависимости между задачами — M
**Проблема:** Задачи независимы. Нельзя сказать "сначала сделай рефакторинг, потом добавь фичу на его основе".
**Решение:** Поле `depends_on: [task-id]` в `QueuedTask`. Задача ждёт, пока зависимости не перейдут в `passed`.

### 3.11 Template-промпты для частых типов задач — S
**Проблема:** Для похожих задач (баг-фикс, рефакторинг, добавление тестов) пользователь каждый раз пишет промпт с нуля.
**Решение:** Шаблоны в `.nightshift.yaml` (`prompt_templates`): `bugfix`, `refactor`, `add-tests`, `docs`. Поле `template` в задаче для выбора.

### 3.12 Оценка стоимости запуска — S
**Проблема:** Пользователь не знает, сколько стоит ночной запуск в деньгах (API tokens Claude).
**Решение:** Парсить usage из Claude output (или запрашивать через API), считать примерную стоимость, показывать в `nightshift status`.

## 4. Developer Experience (DX)

### 4.1 `nightshift config show` — S
**Проблема:** Нет команды для просмотра текущей конфигурации в удобном виде. Нужно вручную читать YAML-файлы.
**Решение:** Команда, которая показывает merged view глобального + проектного конфига в виде красивой таблицы.

### 4.2 `nightshift config edit` — S
**Проблема:** Для изменения конфига нужно вручную редактировать YAML. Можно ошибиться в формате.
**Решение:** Команда, которая открывает конфиг в `$EDITOR` и валидирует после сохранения.

### 4.3 Автодополнение shell (completion) — S
**Проблема:** Typer поддерживает shell completion, но оно не настроено.
**Решение:** Добавить `app.command("completion")` или инструкции в `nightshift install` для установки completions.

### 4.4 `nightshift test` — запуск одной задачи в foreground — M
**Проблема:** Для отладки промпта нужно создавать задачу, добавлять в очередь, запускать run. Долго.
**Решение:** `nightshift test --task "Fix the login bug" --project ./my-app` — одноразовое выполнение с выводом в терминал, без записи в очередь и создания PR.

### 4.5 Dry-run с оценкой сложности — S
**Проблема:** `--dry-run` только показывает список задач. Не даёт оценку, сколько времени/ресурсов потребуется.
**Решение:** Показывать estimated_minutes, общее время, предупреждение если превысит max_duration_hours.

### 4.6 Прогресс-бар при выполнении — S
**Проблема:** При `nightshift run` нет визуального прогресса. Непонятно, что происходит.
**Решение:** Rich progress bar: `[2/5] Running "Fix login bug"... 3m elapsed`.

### 4.7 `nightshift diff <task-id>` — S
**Проблема:** Чтобы посмотреть, что сделал Claude для конкретной задачи, нужно переключаться на ветку.
**Решение:** Команда, которая показывает `git diff main...branch` для задачи прямо в терминале.

### 4.8 Валидация `.nightshift.yaml` при загрузке — S
**Проблема:** `load_project_config` молча игнорирует неизвестные поля и не сообщает об опечатках.
**Решение:** Pydantic `model_config = ConfigDict(extra="forbid")` или хотя бы warning при неизвестных ключах.

## 5. UX

### 5.1 Цветной вывод ошибок качества — S
**Проблема:** При провале quality gates ошибка выводится plain text. Сложно быстро понять, что пошло не так.
**Решение:** Форматирование gates_msg в Rich: красный для failed, зелёный для passed, таблица с чекмарками.

### 5.2 `nightshift sync --auto` без интерактивных вопросов — S
**Проблема:** `sync` использует `questionary.select` при конфликтах. В автоматическом режиме (cron) это зависнет.
**Решение:** Флаг `--auto` (или `--non-interactive`): при конфликте автоматически `skip`. Проверять `sys.stdin.isatty()`.

### 5.3 Интернационализация CLI-вывода — L
**Проблема:** Все сообщения на английском. Для русскоязычных пользователей может быть неудобно.
**Решение:** Не приоритетно, но можно вынести строки в отдельные файлы и поддержать i18n через `gettext`.

### 5.4 Группировка вывода `tasks list` по проектам — S
**Проблема:** `nightshift tasks list` показывает плоский список. При 20+ задачах из разных проектов трудно ориентироваться.
**Решение:** Группировать по project_path с заголовками. Добавить `--group-by project|status|priority`.

### 5.5 Умная сортировка задач — S
**Проблема:** Сортировка только по приоритету. Нет учёта estimated_minutes, количества попыток, давности добавления.
**Решение:** Конфигурируемая стратегия сортировки: по умолчанию `priority, -added_at`. Опция `--sort` в CLI.

## 6. Архитектура и расширяемость

### 6.1 Plugin hooks (lifecycle events) — L
**Проблема:** Плагины могут только добавлять source-адаптеры. Нельзя добавить hook на "перед выполнением", "после PR", "при ошибке".
**Решение:** Event system: `on_task_start`, `on_task_complete`, `on_task_fail`, `on_run_complete`. Entry point `nightshift.hooks`. Позволит реализовать нотификации, метрики, кастомные действия как плагины.

### 6.2 Миграция с YAML на SQLite для очереди — L
**Проблема:** `tasks.yaml` — single file, линейный поиск по O(n), нет индексов, нет concurrent access. При сотнях задач будет медленно.
**Решение:** SQLite с `aiosqlite`. Атомарные транзакции, индексы по status/project/priority. Миграция при первом запуске.

### 6.3 Конфигурация через environment variables — S
**Проблема:** Все настройки только через YAML. В CI/CD и Docker удобнее через env vars.
**Решение:** Pydantic settings: `NIGHTSHIFT_MAX_PRS=5`, `NIGHTSHIFT_SCHEDULE_TIME=03:00`. Env vars override YAML.

### 6.4 Типизированный конфиг для `claude_system_prompt` — S
**Проблема:** `claude_system_prompt` — просто строка. Нет структуры.
**Решение:** Поддержать ссылку на файл: `claude_system_prompt: file://CLAUDE.md`. Удобнее редактировать, можно шарить между проектами.

### 6.5 Вынести git_ops в отдельный абстрактный слой — M
**Проблема:** `git_ops.py` содержит и git-операции, и `gh pr create`. Тестировать сложно: всё через subprocess.
**Решение:** Протокол `VCSProvider` с реализацией `GitProvider`. Упростит тестирование (mock) и потенциальную поддержку других VCS.

### 6.6 Разделить `create_pr` на VCS и платформу — S
**Проблема:** `create_pr` в `git_ops.py` напрямую вызывает `gh`. Привязка к GitHub. Для GitLab или Bitbucket нужно будет дублировать логику.
**Решение:** Абстракция `PlatformProvider` с реализациями `GitHubProvider`, `GitLabProvider`. Определять по remote URL.

## 7. Тестирование

### 7.1 Integration tests с реальным git-репо — M
**Проблема:** Тесты `test_git_ops.py` и `test_runner.py` мокают subprocess. Реальные git-сценарии (merge conflicts, detached HEAD) не покрыты.
**Решение:** Pytest fixtures, создающие временные git-репо с коммитами. Тестировать prepare_repo, create_branch, cleanup на реальных репо.

### 7.2 Тесты для CLI-команд — M
**Проблема:** Есть тесты для `sync_cmd` и `tasks_cmd`, но нет для `run`, `doctor`, `install`, `init`.
**Решение:** Использовать `typer.testing.CliRunner` для тестирования всех команд. Мокать внешние зависимости.

### 7.3 Property-based tests для task_queue — S
**Проблема:** CRUD-операции в `task_queue.py` тестируются только happy path.
**Решение:** Hypothesis для генерации произвольных задач и последовательностей CRUD-операций. Проверять инварианты: нет дупликатов ID, save+load = identity.

### 7.4 Тесты для source-адаптеров с recorded HTTP — M
**Проблема:** GitHub/YouTrack/Trello источники требуют реальных API-токенов для тестирования.
**Решение:** `pytest-recording` (VCR.py) для записи и воспроизведения HTTP-ответов. Один раз записать, потом тесты работают offline.

### 7.5 Mutation testing — S
**Проблема:** Code coverage не гарантирует качество тестов.
**Решение:** `mutmut` или `cosmic-ray` для проверки, что тесты действительно ловят баги. Цель: > 80% mutation score для core-логики.

## 8. Производительность и масштабируемость

### 8.1 Ленивая загрузка задач — S
**Проблема:** `load_tasks()` загружает ВСЕ задачи из `tasks.yaml` при каждой операции (даже для `get_task` по ID). При 1000+ задач будет медленно.
**Решение:** Как минимум — кэширование в рамках одного вызова. В идеале — переход на SQLite (см. 6.2).

### 8.2 Streaming вывода Claude — M
**Проблема:** `subprocess.run` с `capture_output=True` буферизирует весь вывод. Для длительных задач (30+ минут) — нет прогресса, большое потребление памяти.
**Решение:** `subprocess.Popen` с чтением stdout/stderr в реальном времени. Писать в лог-файл потоково. Показывать последние строки в progress bar.

### 8.3 Инкрементальный pytest baseline — S
**Проблема:** Baseline-тесты запускают ВЕСЬ тестовый набор. Для больших проектов (1000+ тестов) это 5-10 минут дважды за задачу.
**Решение:** `pytest --co -q` для подсчёта тестов без выполнения (baseline). Или кэширование baseline между задачами в рамках одного run.

## 9. Документация и онбординг

### 9.1 README с quick start — S
**Проблема:** Есть MVP.md, но нет README.md с quick start для новых пользователей.
**Решение:** README с: что это, установка через `pip/pipx`, `nightshift init`, первый запуск.

### 9.2 Примеры `.nightshift.yaml` для разных стеков — S
**Проблема:** Нет примеров конфигурации для Python, Node.js, Go, Rust проектов.
**Решение:** Директория `examples/` с готовыми конфигами и пояснениями.

### 9.3 Troubleshooting guide — S
**Проблема:** Когда что-то ломается, нет руководства по диагностике.
**Решение:** Секция в README или отдельный файл: частые ошибки, как читать логи, как перезапустить.
